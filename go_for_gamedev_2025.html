Go Gamedev State 2025.

Go can absolutely be used for gamedev and has been since Go was released in 2012. While the number of golang gamedevs and game projects has slowly grown, I propose three explanations, philosophical, practical, and technical, as to why Go gamedev is not even more popular.

Philosophical

The philosophical explanation is perfectly presented in Rob Pike’s Less is exponentially more. A great article which answers why games should be developed using Go. It is for the same reasons that Go was created.

Moreover, the key point of the article discusses why C++ [game engine] devs do not switch to Go, and it has to do with “Do you think less is more, or less is less?”

I think Go’s world view would naturally lead to an initial focus on the game dev essentials. My own golang hobby engine1 is a wish list of the core functionality that I would love to see supported in pure Go, eg:

  • asset loading - be selective: png for images, glfw for models, etc.
  • linear math library - 64 bit to work with golang math.
  • platform library - think glfw written in Go.
  • gpu api library - start with the Vulkan API generated from the spec.

Overall my experience of creating small games in UnrealEngine and my own golang engine2 has reinforced my belief that the game dev community would benefit tremendously from some “less is exponentially more”.

Practical

The philosophical leads to the first practical consideration. Where are the core game dev devs that understand “less is more” and have the company $$$ support to create and maintain the golang game dev packages? Every few years, like a gopher, I pop my head up and look around. I’m hoping they are there, but I missed finding them. Anyone?

The second practical reason is that coding can be a very small part of the total game dev process. As such the benefits of coding in Go are muted since code can not directly help with:

  • game design
  • level design
  • 3D modelling and animation
  • narrative design and dialog
  • audio design, capture, and editing
  • 2D art and textures

The result is that small games can be coded in any language, including visual scripting languages like Blueprints in Unreal Engine. Coding being a small part of the game dev effort can be an argument for using Go since any extra effort needed for Go is a small part of the overall game release time.

Technical

I think of the technical issue as the golang impedence problem. Currently many golang game dev packages require cgo to interface with non-golang libraries like:

  • Vulkan GPU API
  • Platform API
  • Audio API

I find wrapping native API’s with cgo frustrating. Frustrating to the point that, where possible, it is easier in the long term to port the essential parts of existing open source libraries to Go packages.

Ideally there is no C compiler in the game dev tech stack (I’m spoiled, I know, and don’t forget lazy). Recently I reduced my game engine support to win64 which allowed me to replace cgo with syscalls. Coding became noticably more enjoyable once it was golang only.

My thinking is that the technical impedence problems are solvable. How much time and effort would need be clarified. For example, is it even possible to write a golang Vulkan API layer without wrappers? Maybe. Does it make sense? Worth the effort? Would love it if one just appeared though.

Summary

Go game development is slowly growing but needs more work to become a viable alternative for new game dev coders. Existing technical issues can solved with some practical time, effort and $$$. In the meantime golang game dev as a hobby is fantastic!
